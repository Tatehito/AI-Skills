# 要件定義

## 概要

- このスキルは「なぜ作るのか」「何を作るのか」を明確にします。
- 価値、ユーザー、スコープにのみ集中し、実装方法には触れません。
- 質問は必ずUserAskQuestion機能を使用します。

## Input

- ユーザーからの短いアイデアや問題意識
- 既存の README.md や過去の requirements.md（存在する場合）
- プロジェクトの背景情報やステークホルダーの期待

## Output

- docs/requirements.md

### Output要件

docs/requirements.md には以下を必ず含めること：

- **目的**: なぜこれを作るのか
- **背景・課題**: 現状の問題と解決したいこと
- **想定ユーザー**: 誰が使うのか、どんなシーンで使うのか
- **ユーザーストーリー**: As a ... I want ... so that ... 形式
- **成功条件**: どうなれば成功か（測定可能な指標）
- **非ゴール**: 今回やらないこと、スコープ外のこと

## ルール

- **実装方法・技術・構成には一切触れない**（例: 「React で作る」「API を使う」などは書かない）
- **解決策を具体化しない**（問題と期待結果のみを書く）
- **曖昧な表現を避ける**（「使いやすく」ではなく「3クリック以内で完了できる」など）
- **ユーザー価値を中心に据える**（機能リストではなく、ユーザーが得られる価値を書く）

## 実行手順

このスキルは**対話型**です。各ステップでユーザーに質問し、回答を得てから次に進みます。

### 1. 背景と課題の理解

- ユーザーの発言から「解決したい課題」を抽出する
- 「なぜ今これが必要なのか？」を明確にする
- 現状の問題点と理想の状態のギャップを特定する

### 2. ターゲットユーザーの特定

- 「誰が使うのか？」を具体的に定義する
- ペルソナや利用シーン（コンテキスト）を描く
- ユーザーの課題や動機を言語化する

### 3. ユーザーストーリーの作成

- 「As a [ユーザー], I want [機能], so that [価値]」形式で書く
- 優先度をつける（Must / Should / Could）
- 各ストーリーが解決する課題を明確にする

### 4. スコープの明確化

- 成功条件（どうなれば成功か？）を定義する
- 非ゴール（今回やらないこと）を明示する
- MVPとして最小限で届けるべき価値を特定する

### 5. 要件ドラフトの作成と確認

- 上記の対話内容を元に docs/requirements.md のドラフトを作成する
- ユーザーに内容を提示し、フィードバックを求める

### 6. フィードバック反映と最終化

- ユーザーのフィードバックを反映して requirements.md を更新する
- 必要に応じて再度確認を繰り返す
- 最終版を docs/requirements.md に出力する
